# แบบทดสอบท้ายบทเรียน บทที่ 8: การเก็บข้อมูลในเครื่องและการเชื่อมต่อเครือข่าย API (Quiz)

แบบทดสอบนี้มีวัตถุประสงค์เพื่อประเมินความรู้ความเข้าใจเกี่ยวกับการใช้งาน AsyncStorage ในการบันทึกข้อมูลถาวรประเภทข้อความและออบเจกต์ซับซ้อน, การดึงและส่งข้อมูลด้วย Fetch API (HTTP GET & POST), การตรวจจับสถานะการตอบสนองความสำเร็จผ่าน `response.ok`, และการทำระบบจดจำล็อกอินด้วยโทเค็น

---

## คำถามประจำบทเรียน (10 ข้อ)

### ข้อ 1: ข้อใดอธิบายคุณลักษณะการจัดเก็บข้อมูลของไลบรารี `AsyncStorage` ใน React Native ได้ถูกต้องที่สุด?
ก. บันทึกข้อมูลประเภทอ็อบเจกต์ของภาษารันไทม์ไว้ในคลาวด์เซิร์ฟเวอร์แบบเรียลไทม์  
ข. จัดเก็บข้อมูลแบบถาวรในรูปแบบกุญแจ-ค่า (Key-Value) เฉพาะข้อมูลชนิดข้อความ (String) บนพื้นที่ตัวเครื่องของผู้ใช้  
ค. เก็บข้อมูลไว้ในหน่วยความจำ RAM ซึ่งจะหายไปทันทีเมื่อปิดหน้าต่างแอปพลิเคชัน  
ง. บันทึกรูปภาพและวิดีโอต้นฉบับลงในแกลเลอรีภาพหลักของสมาร์ทโฟน  

### ข้อ 2: เมื่อผู้ใช้ปิดแอปพลิเคชันหรือทำการรีสตาร์ทตัวเครื่องโทรศัพท์ใหม่ ข้อมูลที่บันทึกไว้ใน `AsyncStorage` จะได้รับผลกระทบอย่างไร?
ก. ข้อมูลจะถูกเข้ารหัสสุ่มและย้ายไปฝากไว้ที่เซิร์ฟเวอร์ของ Expo  
ข. ข้อมูลจะคงอยู่เหมือนเดิมอย่างปลอดภัย เนื่องจากข้อมูลถูกเก็บถาวรลงในไดรฟ์ของระบบปฏิบัติการเครื่อง  
ค. ข้อมูลจะสูญหายทันทีเนื่องจาก AsyncStorage ไม่รองรับหน่วยความจำระยะยาว  
ง. ข้อมูลจะกลายเป็นสัญลักษณ์เครื่องหมายคำถามทั้งหมดจนกว่าจะเปิดสัญญาณ WiFi  

### ข้อ 3: หากต้องการเขียนทับบันทึกหรือส่งชุดข้อมูลสตริงใหม่ไปเก็บใน AsyncStorage ผู้พัฒนาแอปพลิเคชันต้องเรียกใช้งานเมธอดใด?
ก. `AsyncStorage.getItem`  
ข. `AsyncStorage.removeItem`  
ค. `AsyncStorage.setItem`  
ง. `AsyncStorage.clear`  

### ข้อ 4: ข้อใดคือพฤติกรรมที่เกิดขึ้นหากคุณส่งออบเจกต์ผู้ใช้งาน เช่น `const user = { name: "สมชาย", age: 30 };` เข้าไปที่คำสั่ง `AsyncStorage.setItem('user_data', user)` โดยตรงโดยไม่มีการแปลงรูปแบบข้อมูล?
ก. ระบบจะบันทึกข้อมูลออบเจกต์ได้สมบูรณ์และใช้งานได้ปกติ  
ข. ระบบจะเกิดข้อผิดพลาดในการทำงาน หรือข้อมูลจะถูกแปลงกลายสตริงข้อความแทนที่ตัวตนจริงเป็นค่า `"[object Object]"` ซึ่งไม่สามารถนำกลับมาแกะข้อมูลด้านในได้  
ค. ระบบจะอัปโหลดข้อมูลตัวแปรนี้ขึ้นสู่เซิร์ฟเวอร์ของเฟซบุ๊กโดยอัตโนมัติ  
ง. ระบบจะเปลี่ยนสไตล์หน้าจอปุ่มสัมผัสของเครื่องเป็นสีแดงเพื่อเตือนภัย  

### ข้อ 5: วิธีการและเครื่องมือคู่ใดที่เหมาะสมที่สุดในการบันทึกข้อมูลออบเจกต์ข้ามหน้าด้วย AsyncStorage เพื่อไม่ให้สูญเสียคุณสมบัติข้อมูลลูกด้านใน?
ก. ใช้ `JSON.stringify(object)` แปลงสตริงก่อนจัดเก็บ และใช้ `JSON.parse(string)` แกะกลับเป็นออบเจกต์ตอนดึงข้อมูลกลับมา  
ข. ใช้คำสั่ง `toString()` ในการบันทึก และใช้ `Number()` ในการเรียกรับค่ากลับคืนมา  
ค. ใช้ระบบ `AsyncStorage.clear()` ทั้งก่อนเขียนและหลังเขียนข้อมูลใหม่  
ง. ไม่จำเป็นต้องแปลงค่าใด ๆ เพียงใส่คำสั่ง `await` คุมไว้ระบบเนทีฟจะสืบประเภทให้เอง  

### ข้อ 6: เมื่อเรียกคำสั่ง `fetch()` เพื่อเชื่อมข้อมูลผ่านอินเทอร์เน็ต หากเซิร์ฟเวอร์หลังบ้านส่งรหัสสถานะข้อผิดพลาด เช่น `404 Not Found` หรือ `500 Internal Server Error` กลับมา ตัวกลไก Promise ของ Fetch จะมีพฤติกรรมอย่างไร?
ก. จะสั่ง Reject เข้าสู่บล็อกข้อผิดพลาด `catch` โดยอัตโนมัติ  
ข. จะยังทำงานส่งต่อความสำเร็จ (Resolve) ตามปกติโดยไม่วิ่งไปที่บล็อก `catch` ซึ่งผู้พัฒนาต้องตรวจสอบความสมบูรณ์ผ่านพร็อพส์ `response.ok` ด้วยตนเอง  
ค. จะปิดระบบเครือข่ายของแอปเพื่อป้องกันความเสียหาย  
ง. จะทำลายการเชื่อมต่อสเตตสถานะล็อกอินของผู้ใช้ทั้งหมดทิ้งทันที  

### ข้อ 7: พร็อพส์ (Properties) ตัวใดของ Response Object ที่ตอบกลับจากคำสั่ง `fetch()` ซึ่งระบุค่าสถานะความสำเร็จของเครือข่ายที่มีรหัส HTTP Status อยู่ในช่วง 200 ถึง 299?
ก. `response.statusText`  
ข. `response.headers`  
ค. `response.ok`  
ง. `response.bodyUsed`  

### ข้อ 8: การเขียนคำสั่งส่งข้อมูลแบบ HTTP POST ด้วยคำสั่ง Fetch API ข้อใดระบุพารามิเตอร์การจัดส่งข้อมูลในช่อง `fetch(url, options)` ได้ถูกต้องและครบถ้วนตามหลักการพัฒนา?
ก. `options: { method: 'POST', headers: { 'Content-Type': 'application/json' }, body: JSON.stringify(data) }`  
ข. `options: { method: 'GET', headers: { 'Accept': 'text/plain' }, body: data }`  
ค. `options: { method: 'DELETE', body: data.toString() }`  
ง. `options: { method: 'POST', body: data }`  

### ข้อ 9: ในขั้นตอนจัดการการตอบสนองผู้ใช้ขณะที่ฟังก์ชัน `fetch()` กำลังวิ่งทำงานอยู่บนเน็ตเวิร์ก ข้อใดคือแนวปฏิบัติที่ดีในการคุมสเตตเพื่อความพึงพอใจของผู้เรียนหรือผู้ใช้แอป?
ก. ตั้งค่าสเตต `isLoading` เป็น `true` ก่อนเรียกเฟทช์ และสลับเป็น `false` ในบล็อก `finally` พร้อมแสดงคอมโพเนนต์ `ActivityIndicator` ระหว่างรอสายข้อมูล  
ข. ปิดปุ่มกดไม่ให้สัมผัส และห้ามแสดงความคืบหน้าใด ๆ บนจอจนกว่าแอปจะค้าง  
ค. แสดงกล่องข้อความเตือนภัยสีเหลืองกะพริบบนขอบหน้าต่างจอมือถือ  
ง. บังคับย้อนกลับหน้าแรกสุดของแอปพลิเคชันทุกครั้งที่ระบบกำลังรอโหลดข้อมูล  

### ข้อ 10: ในระบบจัดการจดจำสิทธิ์การล็อกอิน (Auth Token Caching) หากเราต้องการให้โทรศัพท์มือถือทำการเช็คประวัติชื่อล็อกอินในเครื่องเพื่อกระโดดข้ามหน้ากรอกแบบฟอร์ม เราต้องเรียกเขียนโค้ดตรวจสอบโทเค็นในจุดใดของแอป?
ก. เขียนตรวจในฟังก์ชัน `onPress` ของปุ่มล็อกเอาต์  
ข. เขียนตรวจในบล็อกสเตต `useEffect` ของหน้าต่างแสดงผลแรกสุด (Root Component) เพื่ออ่านค่าคีย์โทเค็นจาก AsyncStorage ตอนเปิดแอปขึ้นมา  
ค. เขียนระบุซ้อนอยู่ในสไตล์ชีทภายนอกเพื่อให้คอมไพเลอร์เช็คสี  
ง. เขียนส่งในพารามิเตอร์ `body` ของ POST request แรกสุดตอนสลับสี

## เฉลยคำตอบและคำอธิบายโดยละเอียด

### เฉลย ข้อ 1: ตอบ ข.
* **เหตุผลประกอบ:**
  * **ก. ผิด:** AsyncStorage เก็บข้อมูลบนเครื่องส่วนตัวปลายทาง (Local Storage) ไม่ใช่ฐานข้อมูลระบบ Cloud
  * **ข. ถูก:** เป็นระบบจัดเก็บบนเครื่องเก็บแบบคีย์-ค่า และทำงานรองรับเฉพาะประเภทข้อมูล String เท่านั้น
  * **ค. ผิด:** เป็นระบบจัดเก็บถาวร (Persistent) ข้อมูลจะไม่สูญหายเมื่อปิดแอปพลิเคชัน
  * **ง. ผิด:** ไม่ได้มีสิทธิ์เข้าถึงหรือบันทึกรูปลงคลังภาพส่วนกลางของแอปกล้องโดยตรงโดยไม่มีสิทธิ์พิเศษ

### เฉลย ข้อ 2: ตอบ ข.
* **เหตุผลประกอบ:**
  * **ก. ค. และ ง. ผิด:** ข้อมูลไม่ได้ถูกส่งไปภายนอก และไม่สูญหายเมื่อไฟดับหรือรีสตาร์ทตัวเครื่อง
  * **ข. ถูก:** หน้าที่ของ AsyncStorage คือการเขียนไฟล์ข้อมูลลงบนฮาร์ดไดรฟ์ภายในเครื่องแบบถาวร ข้อมูลจึงยังคงปลอดภัยอยู่ครบถ้วน

### เฉลย ข้อ 3: ตอบ ค.
* **เหตุผลประกอบ:**
  * **ก. ผิด:** `getItem` ใช้สำหรับอ่านข้อมูล  
  * **ข. ผิด:** `removeItem` ใช้สำหรับลบข้อมูลชิ้นนั้น ๆ  
  * **ค. ถูก:** `setItem` ทำงานรับค่าพารามิเตอร์ Key และ Value เพื่อบันทึกเขียนข้อมูลทับหรือบันทึกใหม่  
  * **ง. ผิด:** `clear` ใช้ล้างลบข้อมูลทุกชิ้นในคลัง

### เฉลย ข้อ 4: ตอบ ข.
* **เหตุผลประกอบ:**
  * **ก. ผิด:** ข้อมูลออบเจกต์จะไม่ถูกเก็บไว้สมบูรณ์หากไม่ผ่านการแปลงค่าก่อน  
  * **ข. ถูก:** เนื่องจาก AsyncStorage รับค่าสตริงเท่านั้น หากส่ง JavaScript Object เข้าไปโดยไม่ใช้เครื่องมือแปลง ค่าจะถูกแปลงเป็นรูปข้อความ `"[object Object]"` ส่งผลให้ตอนดึงข้อมูลกลับมา ข้อมูลโครงสร้างคุณสมบัติภายในจะหายไปทั้งหมดใช้งานไม่ได้
  * **ค. และ ง. ผิด:** ไม่มีส่วนต่อพ่วงความเชื่อมโยงกับเฟซบุ๊ก หรือการเปิดไฟแอปเตือนภัยทางสี

### เฉลย ข้อ 5: ตอบ ก.
* **เหตุผลประกอบ:**
  * **ก. ถูก:** เป็นมาตรฐานสากลในการจัดเก็บออบเจกต์บนระบบ AsyncStorage โดยแปลงเป็นสตริงรูปแบบ JSON ก่อนบันทึก และถอดสตริงข้อความกลับมาเป็น JavaScript Object ใช้งานผ่าน `JSON.parse`
  * **ข. ค. และ ง. ผิด:** การทำ `toString` ทั่วไปไม่รองรับโครงสร้างซับซ้อน และ `clear` จะล้างข้อมูลทิ้งทั้งหมด

### เฉลย ข้อ 6: ตอบ ข.
* **เหตุผลประกอบ:**
  * **ก. ผิด:** Fetch จะสั่ง Reject สัญญาณเฉพาะกรณีเกิดปัญหาการเชื่อมต่อทางกายภาพล้มเหลว (เช่น เน็ตตัด สัญญาณหลุด หรือไม่พบ DNS โดเมน) เท่านั้น
  * **ข. ถูก:** รหัสข้อผิดพลาดประเภท HTTP Status (เช่น 404, 500) จากเครื่องเซิร์ฟเวอร์ยังถือว่ารับส่งสำเร็จในทางระบบเชื่อมต่อ เอนจิ้น Fetch จึงยัง Resolve คืนผลลัพธ์มาให้ เราจึงจำเป็นต้องตรวจเช็คค่าความสำเร็จผ่าน `response.ok` ด้วยตนเองเพื่อความปลอดภัย
  * **ค. และ ง. ผิด:** ไม่ได้ปิดระบบเน็ตเวิร์ก หรือทำลายข้อมูลโทเค็นในเครื่องโดยอัตโนมัติ

### เฉลย ข้อ 7: ตอบ ค.
* **เหตุผลประกอบ:**
  * **ก. ข. และ ง. ผิด:** พร็อพส์อื่น ๆ ให้ข้อมูลประเภทข้อมูลหรือส่วนหัว แต่ไม่ได้เก็บสถานะสำเร็จเป็น Boolean
  * **ค. ถูก:** `response.ok` เป็นคุณสมบัติประเภท Boolean (true/false) ที่ตัวระบบ Fetch เตรียมให้ตรวจสอบสถานะว่ารหัส HTTP Code ตอบกลับมาจากเซิร์ฟเวอร์เรียบร้อยดีหรือไม่ (มีค่าเป็น true เมื่อรหัสอยู่ในช่วง 200-299)

### เฉลย ข้อ 8: ตอบ ก.
* **เหตุผลประกอบ:**
  * **ก. ถูก:** โครงสร้าง HTTP POST ที่ถูกต้องจำเป็นต้องเปลี่ยน `method` เป็น `'POST'` เติม Headers แจ้งเซิร์ฟเวอร์ว่าเราส่งฟอร์มข้อมูลชนิด JSON และแปลงชุดข้อมูลให้กลายเป็น String ในคีย์ `body`
  * **ข. ค. และ ง. ผิด:** ผิดข้อกำหนดด้านวิธี และไม่มีการตั้งค่า Headers ครอบสตรีมสตริงที่ถูกต้อง

### เฉลย ข้อ 9: ตอบ ก.
* **เหตุผลประกอบ:**
  * **ก. ถูก:** การสลับค่าสเตต `isLoading` ตั้งค่าเพื่อควบคุมและเรนเดอร์ไอคอน ActivityIndicator ระหว่างดาวน์โหลดเป็นเทคนิคมาตรฐานในการแจ้งความเคลื่อนไหวเพื่อไม่ให้เครื่องดูค้างนิ่ง
  * **ข. ค. และ ง. ผิด:** ไม่ควรปิดความเคลื่อนไหวเงียบโดยไม่มีปฏิกิริยา และไม่ควรกระพริบเตือนตื่นตระหนก หรือบังคับย้ายหน้าจอผู้ใช้ไปมา

### เฉลย ข้อ 10: ตอบ ข.
* **เหตุผลประกอบ:**
  * **ก. ผิด:** ปุ่มล็อกเอาต์มีจุดประสงค์เพื่อทำลายโทเค็นทิ้ง ไม่ใช่ตรวจสอบสิทธิ์เข้าใช้งาน
  * **ข. ถูก:** วงรอบชีวิตการเรนเดอร์เริ่มต้นของคอมโพเนนต์จะกระตุ้นคำสั่ง `useEffect` เป็นด่านแรกสุด การตรวจค้นหาข้อมูล `user_auth_token` ในจุดนี้ จะทำให้เราสลับเปลี่ยนหน้าจอได้ถูกต้องทันทีตั้งแต่แอปเปิดหน้าแรก
  * **ค. และ ง. ผิด:** ไฟล์สไตล์ชีทภายนอกควบคุมเฉพาะความสวยงามดีไซน์ และ POST request ส่งข้อมูลหาเซิร์ฟเวอร์ไม่ใช่การดึงตรวจค่าในโทรศัพท์
